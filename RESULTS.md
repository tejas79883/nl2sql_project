## Q03 – Show me appointments for last month

**Status:** ✅ PASS

**Generated SQL:**
```sql
SELECT a.id, p.first_name || ' ' || p.last_name AS patient_name,
       d.name AS doctor_name, a.appointment_date, a.status
FROM appointments a
JOIN patients p ON p.id = a.patient_id
JOIN doctors d  ON d.id = a.doctor_id
```

## Q15 – Show the busiest day of the week for appointments

**Status:** ✅ PASS

**Generated SQL:**
```sql
SELECT CASE strftime('%w', appointment_date)
    WHEN '0' THEN 'Sunday'
    WHEN '1' THEN 'Monday'

    WHEN '3' THEN 'Wednesday'
    WHEN '4' THEN 'Thursday'
    WHEN '5' THEN 'Friday'

LIMIT 1;
```

### Known Limitations / Edge Cases

1. **Agent memory is in-process** – `DemoAgentMemory` resets when the server restarts. The `seed_memory.py` script must be re-run each time the server starts if memory persistence is required. In production, replace with a persistent vector store (ChromaDB, Qdrant, etc.).

2. **Revenue join assumption** – Q06 and Q19 join `invoices` to `appointments` via `patient_id`. This gives an approximate revenue attribution per doctor (all invoices for a patient attributed to each doctor they saw). In production, a direct `appointment_id` FK on invoices would give exact per-visit revenue.

3. **Future-dated appointments** – The database may contain a small number of future "Scheduled" appointments depending on when `setup_database.py` was run. These are excluded from status-based queries by design.

4. **No-show percentage** – Q14 returns ~15.8%. The dummy data uses weighted randomness (`weights=[10, 55, 20, 15]` for Scheduled/Completed/Cancelled/No-Show), so this percentage will vary slightly on each database regeneration.
